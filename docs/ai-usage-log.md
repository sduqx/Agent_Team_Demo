# AI 工具使用记录

> 本文档记录 Agent Team Demo 项目开发过程中，AI 工具（Claude Code、ChatGPT 等）在各阶段的具体使用情况，包括关键 Prompt、AI 产出内容和人工审核决策。
>
> ⏰ 记录时间：2026年5月29日 — 6月4日

---

## 一、总体使用概况

| 阶段 | 主要 AI 工具 | 使用方式 |
|------|-------------|---------|
| paperclip 探索 | Claude Code | 代码解读 + 方案设计 |
| v1.0 框架搭建 | Claude Code | 生成代码骨架 |
| v2.0 依赖管理 | Claude Code + ChatGPT | Prompt 优化 + 边缘 case |
| v3.0 ReAct 重构 | Claude Code | 代码精简 + 工具注册表 |
| v4.0 审查机制 | Claude Code | Debug 辅助 + 多后端适配 |
| v5.0 通用化重构 | Claude Code | 代码实现 + 文档编写 |
| 文档编写 | Claude Code + ChatGPT | 生成初稿 + 格式化 |

**核心设计决策（ReAct 循环设计、三阶段流程、依赖管理算法、通用 Worker 参数化设计）均为人工主导。**

## 二、阶段一：paperclip 探索与快速原型

### 2.1 目标

在成熟的 Multi-Agent 框架 paperclip 上快速验证"多 Agent 协作开发"想法的可行性。

### 2.2 关键对话记录

**对话 1：快速理解 paperclip 架构**

> **人工 Prompt：** "帮我分析一下 paperclip 这个 Multi-Agent 框架的核心架构，特别是 Agent 之间的通信机制和任务分发逻辑。我需要在这个框架基础上构建一个固定角色的开发团队。"

> **AI 输出要点：**
> - paperclip 使用 Pub/Sub 模式的消息通信
> - Agent 通过注册 handler 函数响应特定事件
> - 任务分发基于事件触发，没有依赖管理
> - 架构可扩展点：自定义 Agent 子类、自定义消息类型

> **人工审核决策：** AI 对架构的分析基本准确，但发现 paperclip 对固定团队场景的定制成本较高，决定转向自研。

**对话 2：改造方案设计**

> **人工 Prompt：** "在 paperclip 基础上，设计一个开发团队的多 Agent 系统：Lead 负责需求分析，Backend/Frontend/Test/DevOps 四个 Worker 分别负责不同模块。给出具体的代码改造方案。"

> **AI 输出：** 产出了完整的设计文档，包含：
> - 6 个 Agent 类的定义（含 SuperAgent 基类和 5 个角色子类）
> - 消息类型定义（requirement、task、status）
> - 工作流伪代码

> **人工审核决策：** AI 方案中 Agent 间通信过于复杂，实际执行中做了简化。采纳了角色划分和消息类型的思路。

### 2.3 经验总结

- **AI 价值**：快速理解陌生代码库，生成初始设计方案
- **AI 局限**：对框架细节的理解可能有偏差，需要人工验证
- **关键决策**：基于 AI 的分析，决定 paperclip 不适合本场景，转向自研

## 三、阶段二：v1.0 框架搭建

### 3.1 目标

从零搭建多 Agent 协作系统的项目骨架。

### 3.2 关键对话记录

**对话 3：生成初始代码骨架**

> **人工 Prompt：** "我要构建一个多 Agent 协作开发系统。5 个 Agent：Lead（需求分析）、Backend（Flask API）、Frontend（HTML界面）、Test（unittest）、DevOps（部署文档）。用 Python 实现，每个 Agent 独立进程，通过 JSONL 文件通信。请生成项目骨架代码。"

> **AI 输出：**
> - `agent_base.py`：Agent 基类雏形（无 ReAct，直接 LLM 调用）
> - `lead_agent.py`：广播式任务分发
> - 4 个 Worker Agent 文件：各自独立，逻辑固化
> - `shared_context.py`：JSONL 消息总线
> - `run_team.bat/sh`：启动脚本

> **人工修改：**
> - 调整 LLM 调用方式适配 Anthropic 兼容 API
> - 修复消息总线并发读写的竞态条件
> - 添加 `.env` 配置机制

### 3.3 经验总结

- **AI 价值**：快速产出可运行的代码骨架，节省大量重复编码时间
- **AI 局限**：生成的代码需要人工调试（API 调用方式、并发处理等）
- **关键决策**：人工补充了多后端 LLM 支持（`.env` + `ANTHROPIC_BASE_URL`）

## 四、阶段三：v3.0 ReAct 架构重构

### 4.1 目标

将 Agent 从"固化逻辑"升级为"ReAct 自主循环"。

### 4.2 关键对话记录

**对话 4：BaseAgent 基类设计**

> **人工 Prompt：** "我要设计一个通用 Agent 基类，支持：1) ReAct 循环（Observe→Plan→Act）；2) 工具注册表（子类通过 register_tool 注册）；3) Anthropic 原生 Tool-Use。给出 BaseAgent 的核心实现。"

> **AI 输出：** 产出了 `react_loop()` 核心方法 + `register_tool()` 工具注册机制。

> **人工修改：**
> - 重新设计了工具注册表的 input_schema 规范
> - 添加了 `finish_task` 提前退出机制
> - 优化了 Anthropic API 的 tool_use block 解析（兼容 raw_arguments 嵌套）

**对话 5：代码精简**

> **人工 Prompt：** "当前每个 Worker Agent 文件有大量重复的 LLM 调用逻辑，帮我提取到 BaseAgent。子类应该只需要重写 get_system_prompt() 和 _setup_tools()。"

> **AI 输出：** 识别出每个 Worker Agent 中的重复模式，将其统一到 BaseAgent。

> **人工审核：** 基本采纳 AI 重构方案，但人工决定保留各 Worker 的 `max_rounds` 差异化设置。

### 4.3 经验总结

- **AI 对代码重构非常擅长**：能准确识别重复模式并给出重构方案
- **人工主导架构设计**：ReAct 循环的整体设计为人工决策，AI 辅助代码实现
- **代码精简效果显著**：AI 辅助下 v3.0 相比 v2.1 删除 1698 行、新增仅 374 行

## 五、阶段四：v4.0 三阶段工作流

### 5.1 目标

在 LLM 自主决策和人工控制之间建立安全护栏。

### 5.2 关键对话记录

**对话 6：三阶段流程设计讨论**

> **人工 Prompt：** "当前 Lead Agent 分析需求后直接创建任务，如果 LLM 理解偏差会导致整个计划在无人审查的情况下执行。如何设计一个安全机制？"

> **AI 输出建议：**
> - 方案 A：在 submit_plan 后暂停，等待人工输入 y/n 确认
> - 方案 B：Phase 2 展示详细计划，支持 y/n/m（修改后重试）
> - 方案 C：每次单个任务完成后再人工确认下一个

> **人工决策：** 采纳方案 B（三阶段 + m 修改机制），这是安全与效率的最佳平衡点。

**对话 7：多后端 LLM 适配**

> **人工 Prompt：** "当前代码假设使用 Anthropic 原生 API。我需要支持通过 ANTHROPIC_BASE_URL 接入 DeepSeek、智谱等兼容后端。帮我找出所有需要适配的地方。"

> **AI 输出：**
> - 识别出 `anthropic.Anthropic(base_url=...)` 需要从环境变量读取
> - 部分 LLM 将 tool_use 参数包装在 `raw_arguments` 中，需要增加兼容解析
> - `stop_reason` 字段在不同后端可能有差异

> **人工修改：**
> - 人工编写了完整的 `.env` 配置文档和多后端兼容代码
> - AI 辅助 Debug `raw_arguments` 嵌套问题

### 5.3 经验总结

- **AI 适合做方案对比**：提出多个方案并分析优劣，人工做最终决策
- **AI 擅长兼容性分析**：能系统性地找出需要适配的代码位置
- **三阶段流程 100% 人工设计**：审查流程、y/n/m 交互、重试机制全部手动实现

## 六、阶段五：v5.0 通用化重构

### 6.1 目标

将 v4.0 的"固定技术栈 4 角色 Agent"升级为"不限技术栈的通用框架"。

### 6.2 关键对话记录

**对话 8：通用 Worker 设计**

> **人工 Prompt：** "当前 4 个 Worker 各有一个独立文件，技术栈写死在 System Prompt 里。我想设计一个通用 Worker，通过 CLI 参数注入角色和技术栈，让 LLM 自由选择任何语言/框架。给出设计方案。"

> **AI 输出：**
> - `WorkerAgent` 类设计：通过 `--name` 和 `--tech` 参数动态生成 System Prompt
> - System Prompt 示例：声明技术自由度，不限制任何具体语言
> - CLI 参数解析和启动方式

> **人工审核：**
> - 调整 System Prompt 中自验证的优先级和详细程度
> - 添加"必须自验证"的强制要求
> - 设计 Worker 孵化机制（`_spawn_worker` 子进程创建）

**对话 9：三层验证体系设计**

> **人工 Prompt：** "Worker 经常说'我完成了'但实际文件没生成或代码有语法错误。设计一个质量保证机制。"

> **AI 输出：** 提出三层防线概念：
> - Worker 自检：编译/语法检查
> - Lead 验收：检查文件是否存在
> - 人工抽查：最终 Review

> **人工实施：**
> - 人工实现 `run_command` 工具（shell 命令执行 + 超时控制 + 输出捕获）
> - 人工实现 `expectedFiles` 自动提取和验证逻辑
> - 人工设计"文件缺失只警告不阻塞"的柔性策略

**对话 10：文档 & 测试验证指南**

> **人工 Prompt：** "为 v5.0 的测试验证体系写一份详细的验证指南文档。"

> **AI 输出：** 生成了 `测试验证指南.md`，包含三层防线说明、自验证命令示例、排查指南。

> **人工审核：** 基本采纳，人工补充了"可验证的任务描述"撰写示例和快速排查表。

### 6.3 经验总结

- **v5.0 架构决策 70% 人工主导**：通用 Worker 参数化、Lead 孵化机制、三层防线设计均为人工决策
- **AI 辅助实施 30%**：代码实现和文档编写由 AI 辅助加速
- **关键价值**：AI 擅长快速生成文档和样板代码，让人工专注核心设计

## 七、文档编写

### 7.1 AI 辅助方式

所有文档（README、架构文档、Agent 设计文档、开发报告、AI 使用记录）均采用以下工作流：

1. **人工**提供文档大纲和核心内容要点
2. **AI** 根据大纲生成初稿、格式化排版
3. **人工**审核、补充技术细节、调整表述
4. **AI** 做最终格式统一和交叉引用检查

### 7.2 关键对话

**对话 11：README 重构**

> **人工 Prompt：** "当前 README 只包含 v4.0 内容，需要加入 v5.0 的说明。更新 README 使其成为统一的项目入口，包含完整的文档索引导航。"

> **AI 输出：** 重新组织 README 结构，添加 v5.0 章节、文档索引表、技术栈对比。

> **人工审核：** 调整文档索引的层级、确认所有链接有效。

**对话 12：文档目录统一**

> **人工 Prompt：** "将 doc/ 目录和 v5.0/ 目录下的文档统一整理到 docs/ 目录，包括文档整合、交叉引用更新。"

> **AI 输出：** 
> - 制定统一文档目录结构（architecture.md / report.md / agent-design.md / v5.0-*.md）
> - 整合 v4.0 和 v5.0 的 Agent 设计文档
> - 更新所有 cross-reference

> **人工审核：** 确认所有文档内容完整、链接有效。

## 八、AI 使用反思

### 8.1 有效模式

| 模式 | 说明 | 典型案例 |
|------|------|---------|
| **AI 生成骨架 + 人工打磨核心** | AI 产出初稿，人工补充关键决策和细节 | v1.0 代码骨架、文档初稿 |
| **AI 方案对比 + 人工决策** | AI 提出多种方案并分析优劣，人工做最终选择 | v4.0 三阶段流程设计 |
| **AI 代码重构 + 人工验收** | AI 识别重复模式并重构，人工验证逻辑正确性 | v3.0 提取 BaseAgent 基类 |
| **AI 兼容性分析 + 人工适配** | AI 找出适配点，人工编写兼容代码 | v4.0 多后端 LLM 适配 |

### 8.2 无效/低效模式

| 模式 | 问题 | 改进方式 |
|------|------|---------|
| **完全依赖 AI 生成核心逻辑** | AI 对业务理解有限，生成代码需要大量调试 | 先人工设计架构，再让 AI 实现 |
| **一次性提过大需求** | AI 容易遗漏细节，导致返工 | 拆分为小步骤，逐步迭代 |
| **不审查 AI 输出直接采用** | AI 可能出现幻觉或过时 API 调用 | 始终人工审查 AI 的代码建议 |

### 8.3 核心结论

> **AI 是效率倍增器，不是决策替代者。**
> 本项目所有关键架构决策（ReAct 循环、三阶段流程、依赖管理算法、通用 Worker 参数化）均为人工主导设计，
> AI 作为效率工具辅助代码实现、文档编写和 Debug 分析。

在整个开发过程中，AI 工具将开发效率提升了约 **2-3 倍**（特别是在代码生成和 Debug 辅助方面），但对于需要深度理解业务逻辑的架构设计工作，人类的判断力仍然不可替代。
