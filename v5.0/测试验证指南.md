# v5.0 Agent 产出测试验证指南

## 三层验证防线

```
        Worker 自查               Lead 验收             人工抽查
┌─────────┴─────────┐    ┌─────────┴─────────┐    ┌───────┴───────┐
│ run_command 编译   │    │ expectedFiles     │    │ Phase 2 审查   │
│ list_directory 确认 │ → │ 自动文件存在检查   │ → │ 手动运行查看   │
│ 修复 → 重验证      │    │ 缺失文件告警      │    │ 抽查关键文件   │
└───────────────────┘    └───────────────────┘    └───────────────┘
     防线一 (Worker)        防线二 (自动)            防线三 (人工)
```

---

## 防线一：Worker 自验证

### 工作原理

Worker 在调用 `finish_task` 之前，必须执行自验证流程：

1. **`list_directory`** — 确认所有承诺文件已产出
2. **`run_command`** — 编译检查 / 语法检查 / 运行测试
3. 如果自检不通过 → 修复 → 重新验证 → 通过后再 `finish_task`

### Worker 支持的自验证命令示例

| 语言 | 编译/语法检查 | 测试运行 |
|------|-------------|---------|
| Python | `python -m py_compile xxx.py` | `python -m pytest tests/ -v` |
| Node.js | `node -c xxx.js` | `npm test` |
| Go | `go build ./...` | `go test ./...` |
| Java | `javac xxx.java` | `java -cp . org.junit.runner.JUnitCore XXXTest` |
| Rust | `cargo check` | `cargo test` |
| TypeScript | `npx tsc --noEmit` | `npx jest` |
| Flutter | `flutter analyze` | `flutter test` |

### 正确的 Worker 工作流

```
收到任务描述（含"产出文件: a.py, b.py"）
  → write_file a.py
  → write_file b.py
  → list_directory .       ← 确认文件存在
  → run_command "python -m py_compile a.py"  ← 语法检查
  → run_command "python -m py_compile b.py"
  → [如果报错] 修复代码 → 重复验证
  → [全部通过] finish_task "已完成 a.py 和 b.py，语法检查通过"
```

### 验证方法

Worker 窗口的日志可以看到自验证结果：
```
[TOOL] [backend] 调用工具: run_command("python -m py_compile server.py")
  → [exit_code: 0]  ← 通过！
[TOOL] [backend] 调用工具: finish_task("...")
```

## 防线二：Lead 自动验收

### 工作原理

Lead 从任务描述中自动提取 `expectedFiles`（通过 `_parse_expected_files()`），当 Worker 汇报完成时：

1. 检查预期文件是否真实存在
2. 检查文件大小 > 0（非空）
3. 报告缺失文件（警告，不阻塞流程）

### 自动提取规则

系统从任务 description 中自动识别产出文件：
- `产出文件: xxx.py, yyy.js, zzz/` → 提取所有路径
- `dir/file.ext` → 提取所有含扩展名的路径

**示例**：当前 system prompt 中的示例 description
```
产出文件: api_spec.json（API契约）+ server.js（Express入口）
+ db.js（SQLite初始化）+ package.json
```
→ 自动提取：`["api_spec.json", "server.js", "db.js", "package.json"]`

### 验收输出示例

```
[OK] Task #1 (backend) 完成!
[VERIFY] Task #1 产出验证:
  ✅ api_spec.json (1204B)
  ✅ server.js (3210B)
  ✅ db.js (856B)
  ✅ package.json (412B)
[VERIFY] 全部产出文件验证通过 ✅
```

缺失文件时：
```
[OK] Task #2 (frontend) 完成!
[VERIFY] Task #2 产出验证:
  ✅ main.dart (5400B)
  ❌ home_page.dart (缺失或为空)
  ✅ pubspec.yaml (320B)
[WARN] 缺失 1 个产出文件 ⚠️
```

## 防线三：人工抽查

### Phase 2 审查
在计划阶段，用户审查完整的计划（技术栈、角色、任务、依赖关系），可以：
- `y` 批准执行
- `n` 拒绝
- `m` 输入修改意见重新规划

### 执行后抽查
项目完成后，进入项目目录查看产物：
```bash
# 项目产物均在默认的 .project/ 或指定的 --project-dir 目录下
cd .project/
# 检查所有 Agent 生产的文件
dir /s   # Windows
ls -laR  # Linux/macOS
```

## 如何写出"可验证"的任务描述

### ❌ 不推荐
```
description: "实现用户登录功能"
```
→ 无法验证：不知道产出了哪些文件

### ✅ 推荐
```
description: "产出文件: backend/src/auth.py（登录/注册 API）+ 
backend/tests/test_auth.py（单元测试）。
API 端点：POST /api/auth/login（参数: username, password；返回: {token, user}），
POST /api/auth/register（参数: username, password, email）。
包含 JWT token 生成和密码哈希。"
```
→ 可验证：系统知道要检查 `backend/src/auth.py` 和 `backend/tests/test_auth.py`

## 快速排查指南

| 现象 | 可能原因 | 排查步骤 |
|------|---------|---------|
| Worker 没有自验证 | LLM 跳过了验证步骤 | 检查 Worker 窗口日志，看是否调用了 `run_command` |
| 文件验证失败但看起来没错 | 文件路径使用相对路径不对 | 检查 Worker 和 Lead 是否用同一 `project_dir` |
| Worker 一直自检失败 | LLM 生成的代码有 bug | 手动查看产物目录中的代码，按编译命令手动测试 |
| expectedFiles 为空 | 任务 description 格式不规范 | 确保用了"产出文件: xxx"格式，这是自动提取的依据 |
| `run_command` 超时 | 命令执行超过 120 秒 | 检查是否有长时间运行的服务，应在单独窗口启动 |

## 补充建议

### 1. 给 test 角色特殊待遇
test Worker 的 `run_command` 最有价值——它能实际运行测试验证后端/前端代码的正确性：

```
Task #3 → test Worker:
  产出文件: tests/test_api.py
  → run_command "pip install requests && python -m pytest tests/test_api.py -v"
  → 如果测试失败，它应该报告给 Lead
```

### 2. 跨 Agent 集成验证
test Agent 运行测试时，如果依赖后端服务，应该在 `_build_role_tech_desc` 中告知：
```python
"test": "测试工程师。注意：后端服务应该已经启动，测试前用 curl 验证端口可达..."
```

### 3. 如有需要，可升级为"验证不通过则阻塞"模式
当前设计是：文件缺失只做警告，不阻塞下游任务。
如需严格模式，修改 `_handle_mark_task_done` 中的验证逻辑，在 `verify_result["pass"] == False` 时：
- 不调用 `TASK_MGR.update(status="completed")`
- 发送消息给 Worker 要求补充产出
