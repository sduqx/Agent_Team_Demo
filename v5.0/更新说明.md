# v5.0 更新说明

## 概述

v5.0 是一次**架构级重构**，将原来的"写死 Python+HTML 的 4 角色全栈 Agent"升级为**通用多 Agent 协作框架**。核心思路：让 LLM 真正做主——自主选技术栈、自主组团队、自主验证产出。

v5.0 两大核心亮点：

| 亮点 | 一句话 |
|------|--------|
| 🎯 **不限技术栈** | 不再是"只能做 Python Flask 项目"，LLM 自主选择前后端框架、数据库、测试工具 |
| 🛡️ **质量保证** | 三层验证防线：Worker 自验证 → Lead 自动验收 → 人工审查，确保每个 Agent 的产出可靠 |

---

## 近期更新

### 2026-06-04 · ReAct 最大循环次数调整为 50

将 Agent 的 ReAct 循环最大次数从 15/8（Worker 默认15、Lead 默认8）统一调整为 **50 轮**。
目的是给 Worker 足够的自验证和修复机会——新增的自验证流程（编译→测试→修复→重验证）比单纯写文件消耗更多轮次。

| 文件 | 变更前 | 变更后 |
|------|--------|--------|
| `agent_base.py` BaseAgent 默认 | `max_rounds=15` | `max_rounds=50` |
| `worker_agent.py` 默认 + CLI 默认 | `default=15` | `default=50` |
| `lead_agent.py` Lead 默认 | `max_rounds=8` | `max_rounds=50` |

### 2026-06-04 · 三层产出验证体系上线

详见下方 [二、质量保证机制](#二质量保证机制) 章节。

## 一、核心技术栈：从写死到完全自由

### 1.1 问题：v4.0 被钉死在 Python+HTML 上

v4.0 每个 Worker 都是一个独立文件，技术栈硬编码在 System Prompt 里：

```
backend_agent.py  → 永远是 Python Flask
frontend_agent.py → 永远是原生 HTML+CSS+JS
test_agent.py     → 永远是 Python unittest
devops_agent.py   → 永远是 Markdown 文档
```

无论需求是"做一个 Go 微服务"还是"做一个 Flutter App"，都只能产出 Python 代码——因为 Agent 根本不知道自己还能用别的技术。

### 1.2 方案：LLM 自主选择技术栈

v5.0 彻底解耦——**所有 Worker 共用一个 `worker_agent.py` 文件**，角色名和技术栈通过 CLI 参数注入：

```bash
# 不再是死板的 4 个角色——LLM 根据需求决定要什么
python -m src.worker_agent --name backend  --tech "Go Gin 框架 + PostgreSQL 后端开发"
python -m src.worker_agent --name mobile   --tech "Flutter 跨平台移动端开发"
python -m src.worker_agent --name data     --tech "Python Pandas 数据分析"
```

Lead 在分析需求时通过 `submit_plan` 输出完整的技术选型 JSON：

```json
{
  "project_name": "...",
  "tech_stack": {
    "backend": "Node.js Express",
    "frontend": "React 18 + TypeScript",
    "database": "PostgreSQL",
    "reason": "需求要求实时数据推送，Node.js 的 WebSocket 支持更成熟..."
  },
  "tasks": [...]
}
```

用户审查阶段可以看到 LLM 的技术选型和**选择理由**，不满意可以 `m` 修改或 `n` 拒绝重新来。

### 1.3 支持的广度

v5.0 不限制任何技术栈，LLM 可以从它的训练知识中选择最合适的组合：

| 层面 | v4.0（固定） | v5.0（自由选择） |
|------|-------------|------------------|
| 后端 | Python Flask | Flask / FastAPI / Django、Node.js Express/Koa、Go Gin、Java Spring Boot、Rust Actix... |
| 前端 | 原生 HTML+CSS+JS | React / Vue / Angular / Svelte / Flutter / React Native / 微信小程序... |
| 数据库 | 无 | SQLite / PostgreSQL / MySQL / MongoDB / Firebase / Supabase... |
| 测试 | Python unittest | Pytest / Jest / JUnit / Go testing / Rust cargo test... |
| 角色 | 固定4个 | backend / frontend / mobile / test / devops / data / design / security / ai... |

### 1.4 团队按需组建

v4.0 永远派出 backend + frontend + test + devops 四个角色——即使是一个纯后端 API 项目，也会创建多余的 frontend Worker。

v5.0 中 LLM 根据需求自主决定：
- 纯后端 CRUD API → 只需要 **backend + test** 两个角色
- 完整 Web 应用 → backend + frontend + test + devops
- 移动端 App → backend + mobile + test
- 数据管道 → data + devops

## 二、质量保证机制

v5.0 最大的实际需求痛点：**Worker 说"我完成了"，结果文件根本没创建，或者代码有语法错误**。v4.0 对此完全没有验证。

v5.0 建立三层防线，确保每个 Agent 的产出可靠。

### 2.1 整体流程

```
┌──────────────────────────────────────────────────────┐
│                   需求 → Lead 分析                     │
│                        ↓                             │
│              submit_plan（含 tech_stack + 预期产出文件）  │
│                        ↓                             │
│     ┌→ 审查 (y批准 / n拒绝 / m修改) ← 防线三：人工把关    │
│     │                  ↓                             │
│     │        Lead 孵化 Worker 分发任务                  │
│     │                  ↓                             │
│     │  ┌─────────────────────────────────┐           │
│     │  │ Worker 工作流：                    │           │
│     │  │  write_file → 产出代码             │           │
│     │  │  list_directory → 确认文件存在      │ ← 防线一   │
│     │  │  run_command → 编译/语法检查        │   自验证   │
│     │  │  run_command → 运行测试            │           │
│     │  │  [修复 → 重验证，直到通过]          │           │
│     │  │  finish_task → 提交完成            │           │
│     │  └─────────────────────────────────┘           │
│     │                  ↓                             │
│     │  Lead 验收：expectedFiles 自动检查 ← 防线二      │
│     │          文件存在且有内容？                       │
│     │           ✅ 通过 / ⚠️ 缺失告警                  │
│     │                  ↓                             │
│     │         标记完成 → 解锁下游任务                   │
│     │                  ↓                             │
│     └──── 人工抽查 .project/ 产物（可选）               │
└──────────────────────────────────────────────────────┘
```

### 2.2 防线一：Worker 自验证

每个 Worker 的 System Prompt 明确要求 **在 finish_task 之前必须自验证**：

1. **`list_directory`** — 确认所有承诺的产出文件都已创建，且文件大小合理（不是空文件）
2. **`run_command`** — 执行编译/语法检查（如 `python -m py_compile`、`node -c`、`go build`、`tsc --noEmit`）
3. **`run_command`** — 如果有测试文件，运行测试确认通过
4. **修复重来** — 如果验证不通过，修复代码后重新验证，直到全部通过

`run_command` 工具特性：
- 在项目目录中执行 shell 命令
- 超时 120 秒防止卡死
- 返回 exit_code + stdout + stderr，Worker 能看懂报错信息
- 支持任何语言：Python / Node.js / Go / Java / Rust / TypeScript / Flutter...

**这条防线价值最大**：LLM 有很强的"看懂报错并修复"能力，之前只是因为没给工具所以没法用。现在 Worker 写代码 → 跑编译 → 看到报错 → 自动修复，形成了一个完整的自我纠错循环。

### 2.3 防线二：Lead 自动验收

Lead 在收到 Worker 的完成汇报后，自动检查实际产出：

1. **任务创建时**：Lead 从 task description 中自动提取"产出文件: xxx"格式的文件路径，存入 `expectedFiles`
2. **Worker 完成时**：Lead 调用 `TASK_MGR.verify_task_output()` 逐文件检查
3. **结果展示**：
   ```
   [VERIFY] Task #3 产出验证:
     ✅ api/server.py (2340B)
     ✅ api/models.py (980B)
     ❌ api/tests/__init__.py (缺失或为空)
   [WARN] 缺失 1 个产出文件 ⚠️
   ```

**重要设计**：文件缺失只发警告，不阻塞下游任务。原因：
- 文件名可能有拼写差异（`api/server.py` vs `api/app.py`）
- LLM 可能产出了内容相同但路径不同的文件
- 不应该因为一个文件路径匹配不上就卡死整个项目

未来可以升级为"严格模式"，验证不通过则拒绝完成。

### 2.4 防线三：人工审查

两个环节可以人工介入：

| 环节 | 时机 | 操作 |
|------|------|------|
| 计划审查 | Lead 分析完成后、执行前 | `y` 批准 / `n` 拒绝 / `m` 修改需求（带反馈重新规划） |
| 产物抽查 | 项目完成后 | 进入 `.project/` 目录人工查看代码质量和运行效果 |

**审查阶段能看到什么**：
- LLM 推荐的技术栈及选择理由
- 团队角色构成及每个角色的职责
- 完整任务拆分和依赖关系
- 每个任务的预期产出文件

## 三、架构对比

### 3.1 文件数量：5 个专用文件 → 1 个通用框架

```
v4.0:                               v5.0:
backend_agent.py   (写死Flask)       src/lead_agent.py      (动态决策)
frontend_agent.py  (写死HTML/JS)     src/worker_agent.py    (唯一通用Worker)
test_agent.py      (写死unittest)      ├─ --name backend  --tech "Go Gin"
devops_agent.py    (写死Markdown)      ├─ --name frontend --tech "React 18"
lead_agent.py      (模板分析)          ├─ --name mobile   --tech "Flutter"
                                      └─ --name data     --tech "Pandas"
5 个进程手动启动                       只启 Lead，其余自动孵化
```

### 3.2 工作流对比

```
v4.0: [手动启5进程] → 输入需求 → Lead模板分析 → 分发固定Worker → 产出

v5.0: [只启Lead] → 输入需求 → LLM分析 → 人工审查 → 自动孵化Worker → 分发执行 → 自验证 → 自动验收
        ↑                                                     ↑       ↑       ↑
        省去手动启动                                  可拒绝/修改    自动    防线一+二
```

## 四、BaseAgent 工具清单

所有 Agent 共用的 8 个通用工具（BaseAgent 内置，子类无需重复注册）：

| 工具 | 功能 | 说明 |
|------|------|------|
| `write_file` | 写入文件 | 输出到项目目录，自动建子目录 |
| `read_file` | 读取文件 | >5000 字节自动截断 |
| `list_directory` | 列出目录 | 递归列出所有文件及大小 |
| `send_message` | 发消息 | 通过消息总线与其他 Agent 通信 |
| `read_inbox` | 收消息 | 读取并清空收件箱 |
| `ask_user` | 提问 | 遇到需要人类决策时暂停等待输入 |
| `finish_task` | 完成 | 提交总结、退出 ReAct 循环 |
| **`run_command`** | **执行命令** | **v5.0 新增：编译/测试/语法检查，自验证核心** |

## 六、迁移指引

从 v4.0 迁移到 v5.0：

1. **不再需要** `backend_agent.py`、`frontend_agent.py`、`test_agent.py`、`devops_agent.py`（已被 `worker_agent.py` 替代）
2. 只启动 Lead：`python -m src.lead_agent`，其余由 Lead 自动孵化
3. **审查阶段请认真检查**技术选型是否合理，不满意可拒绝（`n`）或带反馈修改（`m`）
4. 调试时可手动启动单个 Worker：
   ```bash
   python -m src.worker_agent --name <角色名> --tech "<技术栈描述>"
   ```

## 七、已知限制

1. **LLM 技术选型不一定合理**：审查阶段务必仔细检查 tech_stack 中的技术选择
2. **Worker 子进程窗口**：Windows 在新 cmd 窗口运行，Linux/macOS 需要 xterm
3. **多项目隔离**：并行项目需要不同 `--project-dir`，否则消息总线混淆
4. **同一 role 多任务**：依赖关系取该 role 第一个任务作为锚点
5. **文件验证不阻塞流程**：expectedFiles 缺失目前仅告警，不阻断下游
